66/*
67 * Although rotating and flipping data expressed as DCT coefficients is not
68 * hard, there is an asymmetry in the JPEG format specification for images
69 * whose dimensions aren't multiples of the iMCU size. The right and bottom
70 * image edges are padded out to the next iMCU boundary with junk data; but
71 * no padding is possible at the top and left edges. If we were to flip
72 * the whole image including the pad data, then pad garbage would become
73 * visible at the top and/or left, and real pixels would disappear into the
74 * pad margins --- perhaps permanently, since encoders & decoders may not
75 * bother to preserve DCT blocks that appear to be completely outside the
76 * nominal image area. So, we have to exclude any partial iMCUs from the
77 * basic transformation.
78 *
79 * Transpose is the only transformation that can handle partial iMCUs at the
80 * right and bottom edges completely cleanly. flip_h can flip partial iMCUs
81 * at the bottom, but leaves any partial iMCUs at the right edge untouched.
82 * Similarly flip_v leaves any partial iMCUs at the bottom edge untouched.
83 * The other transforms are defined as combinations of these basic transforms
84 * and process edge blocks in a way that preserves the equivalence.
85 *
86 * The "trim" option causes untransformable partial iMCUs to be dropped;
87 * this is not strictly lossless, but it usually gives the best-looking
88 * result for odd-size images. Note that when this option is active,
89 * the expected mathematical equivalences between the transforms may not hold.
90 * (For example, -rot 270 -trim trims only the bottom edge, but -rot 90 -trim
91 * followed by -rot 180 -trim trims both edges.)
92 *
93 * We also offer a "force to grayscale" option, which simply discards the
94 * chrominance channels of a YCbCr image. This is lossless in the sense that
95 * the luminance channel is preserved exactly. It's not the same kind of
96 * thing as the rotate/flip transformations, but it's convenient to handle it
97 * as part of this package, mainly because the transformation routines have to
98 * be aware of the option to know how many components to work on.
99 */
100
